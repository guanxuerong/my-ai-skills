# AI Code Review 规范

## 一、CR 目标

基于项目编码规范和设计文档，对本次改动的所有文件进行代码走查，发现潜在问题并自动修复。

## 二、走查维度

### 2.1 代码规范

| 检查项 | 说明 |
|--------|------|
| 命名规范 | 变量/函数/组件命名是否符合 camelCase / PascalCase 规范 |
| 函数风格 | 是否使用箭头函数表达式（func-style: expression） |
| 类型定义 | Props 是否有类型定义，是否有不必要的 any |
| 日期库 | 是否误用 moment（必须用 dayjs） |
| 导入规范 | 是否有重复导入、未使用的导入 |
| 注释规范 | 接口函数是否有 JSDoc，复杂逻辑是否有注释 |

### 2.2 逻辑健壮性

| 检查项 | 说明 |
|--------|------|
| 空值保护 | 是否有可能为 null/undefined 的变量未做保护 |
| 边界条件 | 数组为空、对象为空、字符串为空时是否有兜底 |
| 状态一致性 | 多个相关 state 更新时是否保持一致（如本次 keyWords 冗余问题） |
| 条件渲染 | 条件判断是否完整，是否有遗漏分支 |
| 错误处理 | try/catch 是否覆盖关键异步操作 |

### 2.3 与设计文档一致性

| 检查项 | 说明 |
|--------|------|
| 数据结构 | 实际代码中的数据结构是否与设计文档描述一致 |
| 组件设计 | 组件拆分是否与设计文档的修改文件表格一致 |
| 接口调用 | 接口地址、参数、响应处理是否与接口文档一致 |
| 交互流程 | 用户操作流程是否与设计文档的交互流程图一致 |

### 2.4 性能与可维护性

| 检查项 | 说明 |
|--------|------|
| 冗余 state | 是否有可以从其他 state 派生的冗余状态 |
| 重复代码 | 是否有可以提取为公共函数/组件的重复逻辑 |
| 组件大小 | 单文件是否超过 300 行 |
| 函数长度 | 单函数是否超过 50 行 |
| 硬编码 | 是否有硬编码的值应该提取为常量 |

## 三、走查流程

1. 读取设计文档中的"修改文件表格"，确定本次改动的文件范围
2. 逐文件按上述维度进行走查
3. 发现问题后分为两类：
   - **自动修复**：代码规范类问题（命名、导入、格式等），直接修复
   - **需确认**：逻辑类问题（冗余 state、边界条件、设计不一致等），列出问题和建议方案，等用户确认后修复
4. 走查完成后，在飞书创建 AI 走查文档

## 四、走查文档输出格式

标题格式：`[需求名]_AI走查报告`

| 序号 | 文件 | 问题类型 | 问题描述 | 严重程度 | 处理方式 | 是否已修复 |
|------|------|----------|----------|----------|----------|------------|
| 1 | xxx.tsx | 代码规范 | xxx | 低 | 自动修复 | 是 |
| 2 | xxx.tsx | 逻辑健壮性 | xxx | 中 | 需确认 | 待确认 |

严重程度：P0（影响功能，必须修复）/ P1（潜在风险，建议修复）/ P2（规范问题，可选修复）

## 五、自我学习

CR 规范会根据实际走查中发现的新问题模式持续更新：
- 如果某类问题反复出现，新增到对应维度的检查项中
- 如果发现新的检查维度，新增章节
- 更新时遵循自我学习流程：识别→提出→确认→执行
