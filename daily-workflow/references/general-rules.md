# 通用规则

## 模块专项规范

### 目录结构

模块专项规范存放在 `references/modules/` 目录下，每个模块一个 md 文件。新建模块规范时复制 `_template.md` 并按需填写。

### 加载机制

- **自动匹配**：第一步需求分析时，根据 PRD 内容和涉及的代码目录，对比各模块 md 中的"适用范围"字段自动匹配
- **手动指定**：用户在启动时说"开启 dw + PRD链接，模块：xxx"
- 匹配结果记录到管理文件中，后续步骤自动引用，不重复询问

### 叠加规则

模块专项规范中的各章节会叠加到对应的通用规范上，不会覆盖通用规范：

| 模块规范章节 | 叠加到 | 生效步骤 |
|-------------|--------|----------|
| 额外编码约束 | coding-standards.md | 第三步：代码开发 |
| 额外 CR 检查项 | cr-general 检查维度 | 第五步：AI CR |
| 额外自测场景 | self-testing.md 自测清单 | 第六步：自测 |
| 特殊接口约定 | 接口联调参考 | 第四步：接口联调 |

### 新增模块规范

用户可以随时说"新增模块规范：xxx"，AI 会基于 `_template.md` 创建新的模块规范文件，并询问用户填写各章节内容。

## 文档存放

- 所有文档存放在飞书 `Kiro/` 根目录下对应子目录
- 文档中的链接必须是可点击的飞书链接

## 状态更新

- 每步完成后更新 daily 表格状态

## 日报更新

每次触发关键动作时，在 daily 表格当天日期列对应需求行中追加一条记录，用序号标识，格式如：

```
1.完成需求分析<br/>2.完成设计文档<br/>3.完成开发
```

- 每条记录使用"完成xxx"的表述，不要直接写关键词
- 每条记录之间使用 `<br/>` 换行（飞书表格单元格内换行必须用 `<br/>`，普通换行符无效）
- 用序号标识
- 同一天内多次触发则依次递增序号追加（不覆盖已有内容）
- 常见关键词：需求评审、需求分析、设计文档、开发、接口联调、自测、提测、跟进测试、Bug修复等
- 如当天日期列不存在，先在当前状态列后新增该日期列（列头格式 `MM/DD`）

## 决策原则

- **不确定的事情必须停下来问用户，不要猜**

## 上下文传递

- 每步生成的关键信息（需求名、分支名、文档链接、接口文档地址等）需在后续步骤中自动引用，不要重复询问用户
- 当获取到新的关键信息时（如用户提供接口文档地址），必须立即回填到之前已创建的相关文档中（如设计文档的元信息表格），不要等用户提醒
- 如果会话中断后恢复，先读取 daily_work 表格获取当前需求的状态和已有信息，从断点继续

## 多需求并行

- 同一时间可能有多个需求在不同阶段，每个需求独立跟踪
- 用户切换需求时，通过需求名或分支名定位到对应的 daily 表格行
- 日报列中同一天可以有多个需求的记录

## 中断与恢复

- 用户可以在任意步骤暂停，后续通过"继续 [需求名]"恢复
- 恢复时优先通过 `git branch --show-current` 获取当前分支名，匹配 daily_work 表格中的分支名列，自动定位到对应需求行
- 如果当前分支名能匹配到需求，则无需用户指定需求名，直接从断点继续
- 如果当前分支不匹配任何需求（如在 main/master 上），则读取 daily 表格的"当前状态"列，询问用户要继续哪个需求

## 自我学习与优化

在流程执行过程中，AI 应主动识别可优化的模式并提出改进建议。

### 触发条件

| 触发场景 | 示例 |
|----------|------|
| 用户主动指出流程问题 | "为什么没自动回填"、"这个状态名不对" |
| 用户提出更好的做法 | "用 RangePicker 不要分开"、"日报格式改成这样" |
| 同一类问题重复出现 | 多次忘记同步更新文档、多次遗漏空值保护 |
| 流程执行遇到工具限制 | 飞书 MCP 不支持电子表格，需要 workaround |
| 规范与实际执行不一致 | 规范写的是"待联调"但语义上应该是"联调中" |
| 修改了规范文件但未同步关联文件 | 改了 SKILL.md 的状态值但没更新 REPORT.md 中的流程图 |

### 询问时机

分两类，避免打断用户正在进行的工作：

**立即询问**（用户发起的，当场响应）：
- 用户主动指出了问题或不合理之处
- 用户提出了更好的做法
- 用户直接要求修改规范

**阶段结束时询问**（AI 自己发现的，攒到确认点再提）：
- 流程执行中发现的优化点（如重复问题模式、工具限制 workaround）
- 规范文档与实际执行不一致
- 在当前步骤的确认点（用户说"确认"之前）一并提出，格式如：
  > 💡 在本步骤执行过程中发现 X 个可优化点：
  > 1. [问题描述] → [建议方案]
  > 2. [问题描述] → [建议方案]
  >
  > 是否需要更新规范？确认后我会同步修改所有相关文件。

### 执行流程

1. **识别**：在执行流程或对话中发现可优化点
2. **提出**：按上述时机向用户描述问题和优化方案，说明影响哪些文件（SKILL.md / references/*.md / README.md / REPORT.md）
3. **确认**：等待用户确认，不要自行修改
4. **执行**：用户确认后，自动更新所有相关文件，确保一致（遵循下方"变更同步规则"）
5. **记录**：如有飞书实践文档（如"AI 驱动前端开发工作流"），也同步更新相关章节

### 变更同步规则

无论是 AI 自动修改还是用户手动修改了以下任何文件：

- `SKILL.md`（流程定义、状态值、步骤描述等）
- `references/*.md`（规范文档中的检查项、模板结构、术语等）

都必须立即检查并同步更新所有引用了相关内容的文件，包括但不限于：

- `README.md`（使用说明中的流程描述、状态列表、维度说明等）
- `REPORT.md`（实践报告中的流程图、状态流转图、提效数据等）
- 其他 references/*.md 中的交叉引用
- 飞书实践文档中的相关章节

不要等用户发现不一致后再补改，变更发生时就应该一并完成所有关联文件的同步。

### 注意事项

- 不要积攒多个优化点一次性提出，发现一个就及时提出
- 优化建议要具体，说明"改什么"和"为什么改"
- 用户拒绝的优化建议不要重复提出

## 飞书文档写作注意事项

### 换行规则

- **表格单元格内换行**：必须使用 `<br/>`，普通换行符（`\n`）在表格单元格内无效，会被合并为一行
- **段落之间**：需要空行分隔，单个换行符会被飞书合并为同一段落
- **callout / grid / column 等组件内**：内容前后需要空行，否则内容可能无法正确渲染
- **lark-table 单元格内**：`<lark-td>` 标签内的内容前后必须空行

### 编辑操作注意

- **优先使用 `replace_range` + 精确定位**，避免 `replace_all` 误伤其他内容
- **表格类内容修改**：用 `selection_by_title` 按章节定位更安全，避免定位到表格内部结构
- **`replace_all` 慎用**：会替换文档中所有匹配项，容易误改不相关的内容（如把所有"待填写"都替换掉）
- **插入内容时**：`insert_before` / `insert_after` 的定位文本要足够长（10-20 字符），确保唯一匹配
- **飞书表格行级定位困难**：飞书云文档中的表格（lark-table 或 Markdown 表格）转换后结构复杂，无法精确定位到某一行。需要更新表格中的某几行时，用 `selection_by_title` 替换整个章节（包含表头和所有行），而不是尝试定位单行
