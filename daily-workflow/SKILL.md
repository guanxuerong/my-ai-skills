# 前端开发工作流 Skill

你是一个前端开发专家。当用户给你一个 PRD 链接时，严格按照以下六步流程执行。

## 前置条件

- 飞书根目录：`Kiro/`（https://swn7zpxv453.feishu.cn/wiki/你的知识库ID ，如子目录不存在则创建）
  - `Kiro/daily_work` — 工作跟踪表格（如不存在则自动创建，并初始化表头：需求名、分支名、当前状态）
  - `Kiro/YYYYMMDD_[需求名]_需求分析/` — 需求分析文档
  - `Kiro/YYYYMMDD_[需求名]_需求分析/[需求名]_设计文档/` — 设计文档
  - `Kiro/周报/` — 周报文档
  - `Kiro/总结/` — 季度/年度总结文档
- 飞书表格：`daily_work`
  - 固定列表头：需求名、分支名、当前状态
  - 日报列（动态）：当前状态列之后，按日期动态追加列，列头格式为 `MM/DD`（如 `04/20`）。每天首次操作时检查是否已存在当天日期列，不存在则新增。每个需求当天的工作内容填写在对应日期列中（如：需求评审、开发设计、开发、联调、跟进测试等）
- 规范文档：位于 `.kiro/skills/daily-workflow/references/` 目录下
  - `requirement-analysis.md`
  - `development-design.md`
  - `coding-standards.md`
  - `code-review.md`
  - `test-submission.md`
  - `general-rules.md`
  - `weekly-report.md`
  - `periodic-summary.md`
  - `self-testing.md`

## 六步流程

### 第一步：需求分析

**触发**：用户提供 PRD 飞书链接

**执行**：
1. 读取 PRD 内容，提取需求名
2. 检查飞书 `Kiro/` 目录下是否存在 `daily_work` 表格，不存在则自动创建并初始化表头（需求名、分支名、当前状态）
3. 在飞书 `Kiro/` 目录创建管理文件，标题格式：`YYYYMMDD_[需求名]_需求分析`
4. 根据需求名自动生成 Git 分支名，格式：`feature/xxx-yyy`（英文短横线命名，总长度不超过30字符）
5. 自动创建 Git 分支并切换：`git checkout -b [分支名]`
6. 在 daily 表格中插入新行：
   - 需求名：超链接，指向刚创建的管理文件 `YYYYMMDD_[需求名]_需求分析`
   - 分支名：自动生成的分支名
   - 当前状态：分析中
7. 读取 `references/requirement-analysis.md`，按规范在管理文件中生成需求分析内容
8. 输出待澄清问题列表，逐条询问用户

**确认点**：
- 用户回答所有待澄清问题后，更新分析文档
- 询问："需求分析是否确认？确认后进入设计阶段"
- 用户确认后，更新 daily 表格状态为"设计中"，进入第二步

**异常处理**：
- PRD 链接无效/无权限 → 停止，提示用户检查
- 无法按规范生成 → 输出已生成部分，标记缺失章节，询问是否继续

### 第二步：设计文档

**触发**：用户确认需求分析后自动执行

**执行**：
1. 读取 `references/development-design.md`，基于确认后的需求分析生成设计文档
2. 在飞书 `Kiro/YYYYMMDD_[需求名]_需求分析/` 目录创建文档，标题格式：`[需求名]_设计文档`

**确认点**：
- 输出设计文档后，询问："设计文档是否确认？确认后开始代码开发"
- 用户确认后，更新 daily 表格状态为"开发中"，进入第三步

**异常处理**：
- 生成的 spec 缺少必要章节 → 停止，列出缺失章节，询问是否补充或继续
- 用户提出修改意见 → 更新设计文档，再次确认

### 第三步：代码开发

**触发**：用户确认设计文档后自动执行

**执行**：
1. 读取 `references/coding-standards.md`
2. 确认当前已在第一步创建的 Git 分支上（如不在则切换过去）
3. 询问用户是否已有接口文档：
   - **有接口文档**：用户提供接口文档地址，基于真实接口进行开发
   - **无接口文档**：接口调用部分使用临时 Mock 逻辑（统一放在独立的 mock/临时文件中，添加 `// TODO: 接口联调时替换` 注释标记），后续在第四步联调时替换
4. 基于设计文档生成代码
5. 输出完成清单

**决策点**（遇到以下情况必须停下来询问用户）：
- 第三方库的选择（有多个候选时）
- 涉及修改已有逻辑的重构
- 性能优化方案有 trade-off 时
- 任何不确定的业务逻辑

**完成检查**：
- ESLint 检查通过
- 核心功能手动验证通过
- 无 console.log 等调试代码

**变更同步**：
- 开发过程中如果用户要求调整了实现方案（如组件选型、交互方式、数据结构等与设计文档不一致的改动），必须在代码修改完成后立即同步更新飞书设计文档中对应的章节，保持设计文档与实际代码一致

**确认点**：
- 输出完成清单后，询问："代码开发完成，请检查"
- 用户确认后：
  - 如第三步使用了临时接口 → 更新 daily 表格状态为"待联调"，进入第四步
  - 如第三步已对接真实接口 → 更新 daily 表格状态为"AI CR中"，跳过第四步，进入第五步

### 第四步：接口联调

**触发**：用户确认代码开发完成且存在临时接口逻辑时执行

**执行**：
1. 用户提供接口文档地址
2. 读取接口文档，对照代码中所有 `// TODO: 接口联调时替换` 标记
3. 逐个替换临时 Mock 为真实接口调用（请求地址、参数、响应结构）
4. 输出联调变更清单（列出每个替换点：文件路径、接口名、变更内容）

**确认点**：
- 输出联调变更清单后，询问："接口联调完成，请检查"
- 用户确认后，更新 daily 表格状态为"AI CR中"，进入第五步

**异常处理**：
- 接口文档链接无效/无权限 → 停止，提示用户检查
- 接口文档字段与 Mock 不匹配（字段名/类型/结构差异大）→ 列出差异点，询问用户以哪边为准
- 部分接口文档未提供 → 标记未替换的 Mock 接口，询问用户是否先提测已联调部分

### 第五步：AI Code Review

**触发**：代码开发完成（或接口联调完成）后，进入人工自测前自动执行

**执行**：
1. 读取 `references/code-review.md`，按规范对本次改动的所有文件进行走查
2. 走查维度和检查项使用 `cr-general` skill 的定义，额外启用"与设计文档一致性"维度
3. 读取设计文档中的"修改文件表格"，确定走查范围
4. 代码规范类问题直接自动修复，逻辑类问题列出并等用户确认
5. 走查完成后，在飞书 `Kiro/YYYYMMDD_[需求名]_需求分析/` 目录创建文档，标题格式：`[需求名]_AI走查报告`

**确认点**：
- 输出走查报告，逻辑类问题逐条询问用户是否修复
- 修复完成后，自动更新飞书走查报告文档中对应问题的"是否已修复"状态，保持文档与实际修复进度一致
- 用户确认后，更新 daily 表格状态为"待自测"，进入第六步

### 第六步：自测

**触发**：用户确认接口联调（或代码开发已对接真实接口）后执行

**执行**：
1. 读取 `references/self-testing.md`，按规范生成自测清单
2. 基于需求分析和设计文档，结合自测规范中的各维度（用户输入边界、接口参数完整性与类型、ESLint 与代码规范、业务逻辑边界），提取具体测试场景
3. 在飞书 `Kiro/YYYYMMDD_[需求名]_需求分析/` 目录创建文档，标题格式：`[需求名]_自测清单`
4. 自测清单按规范中的表格格式输出，"实际结果"和"是否通过"留空由用户填写

**确认点**：
- 输出自测清单，询问用户逐条确认手动自测结果
- 自测过程中发现的问题及修复方案，必须在修复完成后自动追加记录到飞书自测清单文档的"自测发现问题及修复记录"章节中，不要等用户提醒
- 自测全部通过后，询问："自测完成，是否进入提测阶段？"
- 用户确认后，更新 daily 表格状态为"提测中"，进入第七步

### 第七步：提测

**触发**：用户确认自测完成后执行

**执行**：
1. 读取 `references/test-submission.md`，按模板生成提测文档
2. 在飞书 `Kiro/YYYYMMDD_[需求名]_需求分析/` 目录创建提测文档，标题格式参照模板中的标题格式规范
3. 自动填充可从流程上下文获取的字段（提测名称、设计文档链接、分支名、是否需联调、接口文档、预估影响范围等）
4. 逐个询问用户提供无法自动获取的字段（飞书项目链接、冒烟执行记录、code review 记录地址、备注等）

**确认点**：
- 输出提测文档后，询问："提测文档是否确认？"
- 用户确认后，更新 daily 表格状态为"已提测"，填入完成日期

**后续状态流转**（由用户手动触发更新）：
- 用户说"开始测试"或"测试中" → 更新状态为"测试中"
- 用户说"已上线"或"上线完成" → 更新状态为"已上线"，填入完成日期

读取 `references/general-rules.md`，严格遵守其中所有规则。

## 周报

**触发**：用户说"写周报"、"生成周报"、"本周总结"时触发

**执行**：
1. 读取 `references/weekly-report.md`，按模板生成周报
2. 从 daily_work 表格中提取本周有工作记录的需求，汇总每天的日报内容
3. 在飞书 `Kiro/周报/` 目录创建文档

**确认点**：
- 输出周报后，询问："周报内容是否确认？"
- 用户可补充或修改后确认

## 季度/年度总结

**触发**：用户说"写季度总结"、"Q1总结"、"年度总结"等关键词时触发

**执行**：
1. 读取 `references/periodic-summary.md`，按模板生成总结
2. 询问用户是否有本期 KPI 文档，如有则读取并逐条对照评估
3. 从 daily_work 表格、PRD 文档、Git 记录等自动提取数据
4. 在飞书 `Kiro/总结/` 目录创建文档

**确认点**：
- 输出总结后，询问："总结内容是否确认？"
- 用户可补充亮点、修改措辞后确认

## 自我学习与优化

在流程执行全程中，AI 应主动识别可优化的模式。当遇到以下情况时，主动向用户提出优化建议：

- 用户指出了流程中的问题或不合理之处
- 同一类问题重复出现
- 用户提出了更好的做法
- 流程执行中遇到了工具限制需要 workaround
- 规范文档中的描述与实际执行不一致

**执行流程**：
1. 识别可优化点，向用户描述问题和优化方案
2. 说明影响哪些文件（skill.md / references/*.md / README.md）
3. 等待用户确认，不要自行修改
4. 用户确认后，自动更新所有相关文件，确保一致
5. 如有飞书实践文档，也同步更新相关章节

**规则**：
- 发现一个就及时提出，不要积攒
- 优化建议要具体，说明"改什么"和"为什么改"
- 用户拒绝的建议不要重复提出

详细规则见 `references/general-rules.md` 中的"自我学习与优化"章节。

## 快速触发

用户可以通过以下方式启动本 Skill：
- "开启 dw" + PRD 飞书链接 → 启动新需求七步流程
- "继续 dw" → 通过当前 Git 分支名自动匹配需求，从断点恢复
- "更新 dw" + 状态描述（如"已上线"）→ 手动更新需求状态
- "写周报" / "写季度总结" / "年度总结" → 生成对应文档
