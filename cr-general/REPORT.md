# 🔍 AI 代码走查 Skill：可扩展、可学习的前端 CR 引擎

> 💡 一个 AI 驱动的 React + Antd 5 前端代码走查 Skill，说"CR"就能用，走查规范会随使用自动进化。

---

## 一、这是什么

cr-general 是一个 Kiro Skill，用 AI 对 React + Antd 5 前端代码进行自动化走查。它不是一个固定的检查清单，而是一个可扩展、可学习、会自我进化的走查引擎。

| 🎯 特性 | 说明 |
|---------|------|
| 🔍 即开即用 | 说"CR"+ 文件路径就能开始走查，零配置 |
| 🧩 可扩展 | 支持自定义专项检查，按需启用 |
| 🧠 会学习 | 走查中发现的新问题模式自动沉淀为规范 |

---

## 二、使用方法

### 🚀 独立走查

```
CR src/view/organization/businessSwitch/**/*.tsx
```

### 🔗 需求关联走查

配合 daily-workflow 使用，在开发流程第五步自动触发，从设计文档获取走查范围。

### 📢 触发词

`代码走查` / `code review` / `CR` / `走查`

---

## 三、走查能力

### 📋 基础维度（每次必检，共 5 大类 50+ 检查项）

| 维度 | 关注重点 | 示例 |
|------|----------|------|
| ✅ 代码规范 | 命名、函数风格、类型、导入、格式 | 禁止 function 声明、禁止 moment、路径别名 |
| 🛡️ 逻辑健壮性 | 空值保护、条件判断、错误处理 | JSON.parse 保护、0 值渲染、表单重置一致性 |
| 🌐 接口调用 | 请求时机、loading、竞态、参数类型 | 防重提交、保存后刷新时序 |
| ⚡ 性能与可维护性 | 冗余 state、组件大小、硬编码 | 嵌套三元禁止、console.log 清理 |
| 💡 优化建议(P3) | Hooks 使用、状态设计、代码风格 | useState 滥用、useWatch 风险、内联函数 |

### 🎯 专项检查（按需启用）

| 专项 | 适用场景 |
|------|----------|
| 🔄 Antd 升级 | 版本升级后的 Table/Form/Modal/Select/DatePicker/Upload 重点关注 |
| ⚛️ React 升级 | 17→18 的自动批处理、Strict Mode、事件系统、第三方库兼容性 |
| 🎁 IP 联名套餐 | 套餐价格计算、时间判断、门店选择回显 |
| 📣 营销活动 | 状态流转、时间校验、库存边界 |
| ✏️ 自定义 | 用户描述场景，AI 自动生成检查项 |

### 🏷️ 严重程度

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| 🔴 P0 | 紧急重要，必须现在就修改 | 立即修复 |
| 🟠 P1 | 重要但不紧急，可以稍后改 | 排期修复 |
| 🟡 P2 | 不是很关键，可以以后再改 | 视情况修复 |
| 🟢 P3 | 优化建议，提升代码质量 | 有空再改 |

## 四、核心特性

### 🧩 可扩展性

> 走查维度不是写死的。用户可以随时新增专项检查维度，只需描述业务场景，AI 自动生成对应的检查项。已有的专项也可以根据项目实际情况增减检查项。

### 🧠 自我学习

> 走查过程中发现的新问题模式会自动沉淀：
> - 某类问题反复出现 → 新增到对应维度的检查项
> - 发现新的检查维度 → 新增章节
> - 用户指出遗漏 → 补充并更新规范
>
> 流程：AI 识别 → 提出建议 → 用户确认 → 自动更新 SKILL.md
>
> 每个人用出来的 CR skill 都是最适合自己项目的版本。

### 🔗 与 daily-workflow 集成

> cr-general 同时作为 daily-workflow 七步流程中第五步（AI CR）的走查引擎。两个 skill 共享同一套检查项，维护一份规范，两边同步生效。

## 五、输出示例

走查报告包含：

1. 📊 **走查概览** — 各维度通过/问题数统计 + 整体评估
2. 📝 **问题详情表格** — 文件、类型、描述、严重程度(P0-P3)、修复建议、是否已修复
3. 💡 **AI 建议** — 代码质量建议 + 风险提示
